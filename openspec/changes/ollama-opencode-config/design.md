## Context

OpenCode reads config from `~/.config/opencode/opencode.jsonc` (global user config). The current file only has a `$schema` key — no providers declared. The system already runs Ollama on port 11434 on both hosts (`mary` and `bob`), serving local models (e.g. via `ollama list`). In Emacs, gptel already uses `localhost:11434` as an Ollama backend.

The dotfiles are Nix-managed via Home Manager. The `opencode.json` file at `~/.config/opencode/` should be generated by the Home Manager config, not edited manually.

## Goals / Non-Goals

**Goals:**
- Add an Ollama provider to the global OpenCode config so `/connect` is unnecessary for local usage
- Declare locally available models dynamically so the model list stays current
- Manage everything through the existing Home Manager / Config.txt tangle workflow

**Non-Goals:**
- Not configuring cloud providers (those can be added separately)
- Not modifying OpenCode's TUI theme, keybinds, permissions, or other settings
- Not writing a Home Manager module from scratch if an existing pattern suffices

## Decisions

1. **Config location**: `~/.config/opencode/opencode.json` (`.json` not `.jsonc` — OpenCode accepts both) is generated at Home Manager activation time, not symlinked from the Nix store, because it contains runtime-discovered model data.

2. **Model list strategy**: A Nix `pkgs.writeShellApplication` script (`generate-opencode-config`) queries the Ollama HTTP API (`http://localhost:11434/api/tags`) and injects the model list into a JSON template at Home Manager activation time. A standalone `bin/update-opencode-models` script also exists for manual use (calling `ollama list`). The activation hook runs every `home-manager switch`, keeping the model list fresh.

3. **Provider wiring**: The OpenCode docs specify `@ai-sdk/openai-compatible` as the npm adapter for Ollama, with `baseURL: http://localhost:11434/v1`. This is the standard pattern used by all local-model providers in OpenCode.

4. **No SOPS needed**: The Ollama provider has no API keys — it connects to a local endpoint. No secrets involved.

5. **Home Manager approach**: A full `dpom-opencode` module in `modules/home/opencode.nix` (tangled from `Config.txt`) with `programs.opencode.enable = true` and a `home.activation` hook that generates `~/.config/opencode/opencode.json` at switch time. The module pattern (`dpom-<name>.enable`) matches the rest of the dotfiles convention.

## Risks / Trade-offs

- **Stale model list**: If new models are pulled in Ollama but the OpenCode config isn't regenerated, they won't appear in `/models`. Mitigation: the activation hook runs every `home-manager switch`.
- **Build-time vs runtime**: Nix evaluates configs at build time, so `ollama list` can't run during `nix build`. The activation script runs at switch time (via `home.activation`), which is acceptable.
- **Two scripts**: The module's activation script (uses Ollama HTTP API) and `bin/update-opencode-models` (uses `ollama list` CLI) are independent — the latter is a convenience for manual updates without a full `home-manager switch`.
