## Context

OpenCode reads config from `~/.config/opencode/opencode.jsonc` (global user config). The current file only has a `$schema` key — no providers declared. The system already runs Ollama on port 11434 on both hosts (`mary` and `bob`), serving local models (e.g. via `ollama list`). In Emacs, gptel already uses `localhost:11434` as an Ollama backend.

The dotfiles are Nix-managed via Home Manager. The `opencode.jsonc` file at `~/.config/opencode/` should be generated by the Home Manager config, not edited manually.

## Goals / Non-Goals

**Goals:**
- Add an Ollama provider to the global OpenCode config so `/connect` is unnecessary for local usage
- Declare locally available models dynamically so the model list stays current
- Manage everything through the existing Home Manager / Config.txt tangle workflow

**Non-Goals:**
- Not configuring cloud providers (those can be added separately)
- Not modifying OpenCode's TUI theme, keybinds, permissions, or other settings
- Not writing a Home Manager module from scratch if an existing pattern suffices

## Decisions

1. **Config location**: `~/.config/opencode/opencode.jsonc` is already a Home Manager-managed symlink into the Nix store. Add the `provider.ollama` block there rather than in a per-project `opencode.json`.

2. **Model list strategy**: Use a shell script (`bin/update-opencode-models`) that calls `ollama list` and writes the models block into a JSON fragment. Home Manager can reference this via `builtins.readFile` or the script can be run as a post-switch hook. Alternatively, since Nix evaluation happens at build time, a simpler approach: generate the config with a curated model list and document that users re-run a helper when models change. Given Nix's build-time purity, a postSwitch hook is the most practical approach.

3. **Provider wiring**: The OpenCode docs specify `@ai-sdk/openai-compatible` as the npm adapter for Ollama, with `baseURL: http://localhost:11434/v1`. This is the standard pattern used by all local-model providers in OpenCode.

4. **No SOPS needed**: The Ollama provider has no API keys — it connects to a local endpoint. No secrets involved.

5. **Home Manager approach**: Use `xdg.configFile."opencode/opencode.jsonc"` with `builtins.toJSON` or a template file tangled from Config.txt, rather than a full `dpom-opencode` module, to keep scope minimal.

## Risks / Trade-offs

- **Stale model list**: If new models are pulled in Ollama but the OpenCode config isn't regenerated, they won't appear in `/models`. Mitigation: the helper script can be bound to an `ent` command or a post-switch hook.
- **Build-time vs runtime**: Nix evaluates configs at build time, so `ollama list` can't run during `nix build`. The script approach runs at switch time (via `postSwitch`), which is acceptable.
